## 1. 一句话定位

一个**让策划在引擎之外完成关卡"设计 + 验证 + 序列化"的本地主场**：抽象设计（数值/规则/关系）和空间设计（关卡）在同一个模型里绑定，草图阶段就能验证结构正确性，把"在引擎里从零搭建"这段苦力前移并自动化。

MVP 只兑现这个愿景的**最小可用内核**：AI 生成关卡拓扑 → 人扩展 POI → POI 挂真实体。

## 2. 全局约束 / 工程护栏（贯穿所有单元）

| 约束 | 内容 | 后果 |
|---|---|---|
| **G1 开源** | 项目整体开源。 | 依赖许可证规则见 G2。开源是 IP 放大器，不是负担。 |
| **G2 许可证纪律** | MIT / Apache-2.0 / BSD：自由使用，保留版权声明即可。弱 copyleft（LGPL/MPL）：逐个评估用法。**强 copyleft（GPL/AGPL）：只能作为独立外部程序调用，绝不进代码库。** | 每引入一个依赖先过许可证。Tiled 是 GPL → 只能"读其文件格式 / 调独立程序"，不许代码级整合。 |
| **G3 平台形态** | 本地优先。架构 = **本地 daemon（唯一特权进程）+ web 技术界面 + 外部 CLI 作 AI 引擎**。是否再套原生壳（Tauri 等）是发布期打包选择，MVP 不决定。 | 无云、无后端服务器、无账号系统。界面用 web 技术，可浏览器访问。 |
| **G4 单人** | MVP 单人。 | 不做协作、不做同步、不做多用户数据结构。 |
| **G5 数据归属** | 一个项目 = 本地一个文件夹/库，承载全部数据（设计模型、关卡拓扑、资产引用、文档）。 | 打开项目 = 打开一个本地目录。 |
| **G6 AI 接入** | AI 由 daemon **shell 调用用户配置的外部 CLI agent**（Claude Code / Codex / OpenCode 等）实现。GUI 流程中不内嵌、不索取 API key。 | MVP 只支持 CLI 这一种 AI 接入方式。 |
| **G7 AI 是提议者，不是裁决者** | AI 的一切输出是**针对模型的 typed 提议（proposed change）**，经本人 review 后才落盘。AI 永不直接写模型。 | 这是控制权的支点，也是版本/审计能成立的前提。 |
| **G8 确定性核心** | 数据核心是确定性的：所有写入走唯一通道、记录为 append-only 的 typed 事件日志、状态由日志派生。LLM/AI 只活在"提议层"，绝不进核心。 | 这是撤销、历史、审计、AI 提议流能成立的共同地基。具体存储技术（图 vs 关系）开放，但**必须支持 typed schema + 事件日志**这两个不变量。 |

## 3. 功能全景（六区 + 双核心）

```
            序列化 → 引擎自动读取（统一出口）          ← post-MVP
                 ↑                    ↑
      ┌──────────────────┐  ┌──────────────────────┐
      │ A 搭系统（抽象）  │绑定│ D 关卡（空间）        │   ← 两个核心
      │ 数值/规则/关系    │←→│ D-core 拓扑 / D-full画布│
      └──────────────────┘  └──────────────────────┘
                 ↑  共享同一模型 + 视图 + 版本
      ┌─────────────────────────────────────────┐
      │ B 多视图  C 观测器  E 资产  F 文档（开发入口）│   ← 支撑层
      └─────────────────────────────────────────┘
```

| 区 | 职责 | MVP？ |
|---|---|---|
| A 搭系统 | 定义/编辑实体、数值、关系 | **最小子集 in** |
| B 多视图 | 表/图/看板切换看与改 | out（roadmap） |
| C AI 观测器 | 帮选视图、搭面板、存成工具 | out（roadmap） |
| D 关卡 | D-core 拓扑生成+编辑 / D-full 双档画布 | **D-core in，D-full out** |
| E 资产流水线 | 加工→切分→语义登记→被消费 | out（v1.1） |
| F 文档 | 人话+意图+活引用 | out（roadmap） |
| 开发入口 | 从这里可达项目一切 | 定位，随 MVP 自然体现 |

## 4. MVP 范围（稍宽切）

### 4.1 IN — MVP 必须交付
- **D-core**：AI 生成关卡拓扑（房间/区域节点 + 单双动线 + 标记）、拓扑图渲染、在图上编辑、节点上扩展 POI。
- **A-minimal**：定义实体类型 + 创建实例 + 填值；实体可被 POI 引用。
- **绑定**：POI 能挂一个 A 的实体（"这个房间放这个 Boss"）。
- **项目容器**（G5）、**确定性核心+事件日志**（G8）、**AI-CLI 提议通道**（G6/G7）、**线性撤销/历史**。

### 4.2 OUT — MVP 明确不做（不许实现）
- B 多视图、C 观测器、F 文档。
- **D-full**：自由档画布、精准几何/坐标、tile/像素编辑、序列化导出进引擎。
- E 资产流水线（推迟到 v1.1）。
- 多人/协作/同步、账号、云、后端服务器。
- 版本**分支 / 语义 diff / 合并**（MVP 只有线性撤销+历史）。
- 多种关卡工况（MVP 只有"拓扑"这一种）。

### 4.3 MVP 成功的定义（验收闭环）
> 输入一句"做一个有中央大厅 + 三条支线、其中一条是单向捷径的关卡" → AI 返回 typed 拓扑提议 → 本人 review 后接受 → 拓扑图里出现该结构 → 手动加一个房间、给某节点扩展一个 POI 并挂上一个 Boss 实体 → 全程任意一步可撤销、重启后数据还在。

这条跑通 = 第一个对策划真实减负的可用版本。

## 5. 开发单元分解（MVP，按依赖顺序）

> 每个单元：职责 / 做 / 不做 / 完成标准 / 依赖 / 风险。**建议严格按 U0→U5 顺序构建（walking skeleton）。**

### U0 — 项目容器
- **职责**：新建/打开/保存一个本地项目，承载全部数据。
- **做**：新建项目、打开已有项目、定义项目根目录结构、基本元数据（名字/创建时间/版本号）。
- **不做**：多项目工作区切换、云、账号、导入导出整包。
- **完成标准**：新建项目 → 关闭 → 重新打开，数据完整还在；项目目录结构符合约定。
- **依赖**：无（根）
- **风险**：低

### U1 — 确定性核心 + 事件日志
- **职责**：所有数据写入的唯一通道；状态由 append-only typed 事件派生；支持撤销与历史查看。
- **做**：定义事件结构、串行落盘（单一序列化点）、状态物化（fold events）、撤销/重做、线性历史浏览。schema 演化只支持"加字段/加类型"。
- **不做**：分支、语义 diff、合并、复杂 schema 迁移、分布式/多写。
- **完成标准**：任意一次数据修改都进日志；能撤销到任意历史点再重做；重启后由日志重建的状态与关闭前一致。
- **依赖**：U0
- **风险**：中（核心，但范围已收窄到线性历史，不碰分支/合并）

### U2 — 本体 / 实体定义（A-minimal）
- **职责**：定义实体类型、创建实例、填值。A 区的最小版。
- **做**：内置字段类型（文本 / 数值 / 枚举 / 对其他实体的引用）；定义实体类型；增删改实体实例；填字段值。所有变更经 U1。
- **不做**：关系网络/经济循环建模（A3）、约束规则引擎、本体复杂演化、表格批量编辑。
- **完成标准**：能定义一个"Boss"类型（含 hp 等数值字段）、创建两个实例并填不同值；变更进日志、可撤销。
- **依赖**：U1
- **风险**：低

### U3 — 关卡拓扑模型（D-core 数据层）
- **职责**：关卡拓扑的 typed 数据——节点（房间/区域）+ 带方向的边（单向/双向）+ 标记（出生点/捷径等）+ 节点上的 POI 槽。
- **做**：节点、边（含方向性）、标记、POI 槽；POI 可持有对 U2 实体的引用。变更经 U1。
- **不做**：精确几何/坐标、tile/像素、多种工况、序列化进引擎。
- **完成标准**：能把"5 房间、中央↔三支线双向、一条单向捷径"表达为 typed 数据；某节点的 POI 成功挂上 U2 的一个 Boss 实例。
- **依赖**：U1、U2
- **风险**：低-中

### U4 — 拓扑图渲染 + 编辑（D-core 视图层）
- **职责**：把 U3 渲染成节点连线图（关卡逻辑图），并支持在图上编辑。
- **做**：固定的节点连线渲染原语、单/双动线的视觉区分、增删节点与边、改边方向、点节点扩展/编辑 POI。所有编辑动作产生 U1 事件。
- **不做**：自由涂鸦档、精准几何、多工况、AI 生成的"只读美化示意图"（那是后置的生成式输出）。
- **完成标准**：U3 的 5 房间结构能渲染成图；能手动加房间、连一条单向边、给节点加 POI 并挂实体；所有操作可撤销。
- **依赖**：U3
- **风险**：中（交互编辑器，但限定在拓扑节点连线，不含几何）

### U5 — AI-CLI 提议通道（D-core 的"生成"）★高风险，原型先行
- **职责**：调用外部 CLI agent，把自然语言意图变成针对模型的 typed 提议；本人 review 后落盘。
- **做**：配置/定位外部 CLI（参考 daemon 的 PATH 扫描思路）、组织上下文喂给 CLI、解析其产出为 typed 提议、展示提议的语义 diff、接受→经 U1 落盘 / 拒绝→丢弃。MVP 可同步等待（不要求流式）。
- **不做**：AI 直接写模型（必经 review）、多 agent 编排、GUI 内嵌 key、复杂流式交互。
- **完成标准**：输入"生成中央大厅+三支线+一条单向捷径"，CLI 返回提议，review 接受后 U3 出现对应拓扑（即同事要的"AI 出骨架，我扩展 POI"）。
- **依赖**：U1、U3（产关卡拓扑提议）、U4（可视化验证结果）
- **风险**：中-高 —— 跨进程调用 + 把 LLM 输出可靠解析为 typed 提议，是 MVP 最不确定的一块。**建议先做抛弃式原型验证可行性，再进正式实现。**

## 6. 风险分层

| 层 | 单元/模块 | 处理方式 |
|---|---|---|
| **高（原型先行）** | U5（CLI→typed 提议）、未来的 D-full 双档语义画布 | 先抛弃式原型验证体验/可行性，**不直接进正式架构** |
| **中** | U1 核心、U4 编辑器 | 正常开发，留意范围别膨胀 |
| **低** | U0、U2、U3 | 直接做 |

**D-full（精准⇄自由 + 关卡语义组件 + 涂鸦 AI 提升）是全工作台技术风险最高的一块，MVP 完全不碰。** U4 严格限定为"拓扑节点连线"，不引入几何与自由档。

## 7. 明确推迟的决策（Claude Code 不要就这些回头问）

| 推迟项 | 现状 / 约束 | 何时再定 |
|---|---|---|
| 存储形态（图 vs typed 关系） | 实现细节。约束：必须支持 typed schema + 事件日志（G8） | 构建 U1 时由执行者按约束自行定 |
| 工况间数据是否共享一份 | 只在 D-full（多工况）才有意义；MVP 只有"拓扑"一种表达 | D-full 阶段 |
| 原生封装（Tauri 等）vs 纯 localhost web | 打包选择，界面都是 web 技术，零功能影响 | 临发布 |
| CLI 之外的 AI 接入（本地模型直连等） | MVP = 仅 CLI（G6） | post-MVP |
| 精确几何 / 序列化进引擎的契约格式 | 属 D-full / 统一出口 | D-full 阶段 |
| 多视图、观测器、文档、资产流水线的细节 | 全部 out（§4.2） | 各自被排期时 |

## 8. Post-MVP 路线（仅列方向，不展开）

按价值/依赖大致顺序，**细节待各自排期时再定**：

1. **E 资产流水线**（v1.1）：浏览/预览 → 基础加工（放大/图集合并）→ sprite 网格切片 → **语义登记**（进资产模型，核心价值）→ 被消费（设计 @引用 / AI 语义抽取 / 导出）。3D 复用同一条流水线架构。
2. **统一出口·序列化**：关卡结构 + 数值配置 → 引擎自动读取（你"讨厌手动搭建"的最终兑现）。
3. **D-full 双档语义画布**：精准⇄自由切换、网格吸附/坐标约束、关卡语义组件（房间/门/动线/出生点/锁钥）、AI 把自由涂鸦提升为精准结构、多工况（横版/俯视/拓扑…）。
4. **版本进阶**：设计分支、语义 diff、合并、决策溯源（caused_by / intent / evidence）。
5. **B 多视图**：同一模型的表/图/看板等多投影。
6. **C 观测器**：AI 帮选视图、组装工作面板、面板沉淀为常驻工具。
7. **F 文档**：承载人话与意图，通过**活引用**绑定结构区（解 GDD 腐烂）。
8. **多人/协作**：异步分支优先于实时协同。
