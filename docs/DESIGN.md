# DESIGN.md — 〈WORKBENCH〉 UX / 视觉设计契约

> 配套 `SPEC-v1.md`(功能)、`TECH-SPEC.md`(技术)、`CLAUDE.md`(红线)、`WORKFLOW.md`(协作)。
> 这份定**调子和规则**,不是像素稿。Claude Code 做每个面板时照这份走,保证六个区是一个产品、不是六张皮。

---

## 1. 根调(已定 · 一句话)

> **画布为中心,工具环绕**(Figma / open-design 式)· **深色为主** · **冷性中性**(灰/蓝/黑白,专业工具感) · **shadcn/Radix 打底**。

气质参照:Figma 的画布中心布局 + Linear/open-design 的冷静深色工具感。**不是**花哨、不是暖萌、不是信息轰炸。

---

## 2. 最高约束:单一 design token 源(画布与环绕工具共用一套变量)

这是这份文档最重要的一条,**违反就会出现"两张皮"**。

- 你选了 shadcn/Radix 打底,但 **shadcn 只管表单/面板,不管画布**;画布(拓扑图)用 React Flow / Svelte Flow 的自有视觉。
- 因此 UI 分**两层**,但**必须共享同一套 design token**(同一份 CSS 变量:颜色、间距、圆角、字体、阴影):

| 层 | 谁渲染 | 视觉来源 |
|---|---|---|
| **环绕工具层**(侧栏/面板/属性/AI 提议/菜单/对话框/按钮) | shadcn/Radix | shadcn 主题,但颜色/圆角等**读同一套 token** |
| **画布层**(节点/边/POI/选中态/网格) | React/Svelte Flow | Flow 的节点/边样式**也读同一套 token**,不另起一套配色 |

- **实现纪律**:定义一份 design token(`tokens.css` 或等价),shadcn 主题和 Flow 节点样式**都从它取值**。画布里的节点底色、边色、选中色,必须和环绕面板用同一族灰/蓝,而不是 Flow 默认的那套蓝白。
- 目标:鼠标从侧栏移到画布,**感觉是同一个产品**,像 Figma 那样画布和工具面板浑然一体。

---

## 3. Design token(MVP 起步值 · 可后续微调,但保持单一来源)

> 这些是**起步默认**,不是最终视觉。视觉精修推迟(见 §6)。Claude Code 用这套先把 MVP 跑出来。

- **模式**:深色为主(MVP 先只做深色;浅色推迟)。
- **色彩**:冷性中性。
  - 背景:深灰阶(画布最深、面板次之、卡片再浅一档,用层次区分而非边框堆叠)。
  - 文字:高对比近白 / 次级中灰。
  - 主色:一个克制的冷蓝,只用于**强调/选中/主操作**,不滥用。
  - 语义色:成功/警告/错误用低饱和版本,别刺眼(呼应 CLAUDE 对 wellbeing 无关,但工具要冷静)。
- **字体**:UI 用系统无衬线;**数值/ID/代码用等宽**(数值表、节点 id、AI 提议的 diff 都该等宽)。
- **间距/圆角**:统一节奏(如 4px 基数);圆角中等偏小,工具感而非圆润萌系。
- **层次靠背景色阶 + 轻阴影区分,少用重边框线**(Linear/Figma 的做法)。
- 具体色值/数值由 Claude Code 按 shadcn 深色主题落一套合理默认,**只要满足"单一来源 + 冷性中性 + 深色"**即可,不必我逐个钦定。

## 4. 信息架构:画布中心 + 环绕工具(单窗口,不跳页)

呼应你立的"single pane of glass / 不打断心流"——**一个统一工作区,不是六个互相独立的 tab**。

### MVP 布局(只实现这三块 + 顶栏)
```
┌─────────────────────────────────────────────────────────┐
│ 顶栏:项目名 · 撤销/重做 · (后续)区切换                      │
├──────────────┬────────────────────────────┬──────────────┤
│              │                            │              │
│  (预留)       │     画布:关卡拓扑图          │  右侧栏:      │
│  左侧栏       │     (React/Svelte Flow)     │  属性 / POI   │
│  六区导航位   │     节点=房间 边=单双动线     │  选中节点的    │
│              │                            │  字段、挂的    │
│              │                            │  实体          │
├──────────────┴────────────────────────────┴──────────────┤
│  底部/侧浮:AI 提议面板(自然语言输入 → typed 提议 → review)  │
└─────────────────────────────────────────────────────────┘
```
- **中心永远是画布**(MVP 是拓扑图;未来 D-full 的 2D/3D 关卡、B 的其他视图都进这个中心区)。
- **右侧栏 = 上下文属性**(选中什么,这里编辑什么)。
- **AI 提议面板**:输入意图 → 展示 typed 提议的**语义 diff**(等宽)→ 接受/拒绝。**视觉上要让"提议未落盘"状态一眼可辨**(如待接受的变更用虚线/弱化色),呼应 CLAUDE INV-3(AI 是提议者,接受前不改状态)。

### 六区扩展位(MVP 不实现,但骨架留好)
- **左侧栏**是未来六区导航(A 搭系统 / B 多视图 / E 资产 / F 文档 / D 关卡 / AI)的入口位。
- 原则:新区**进同一个窗口的中心区或侧栏**,而不是新开独立页面——保持不跳页、不断流。

## 5. MVP 核心交互动线(只定这几条,其余随区生长)

1. **AI 出骨架 → 人扩展**(同事的核心诉求):AI 提议面板输入"中央大厅+三支线+一条单向捷径" → 画布出现待接受的拓扑(弱化态)→ 接受 → 成为实体节点 → 人在节点上扩展 POI、挂实体。
2. **拓扑编辑**:画布上加/删节点、连边、改边方向(单向/双向视觉可区分)、点节点开右侧栏编辑 POI。每个动作可撤销(顶栏撤销/重做)。
3. **POI 挂实体**:在右侧栏给 POI 选一个 U2 定义的实体(如某 Boss),建立引用。
4. **状态可辨**:已落盘 vs AI 待接受 vs 选中 —— 三种状态视觉上清晰区分。

交互基调:**键盘友好**(参照 Linear),常用操作有快捷键;但 MVP 不必全做,先保证鼠标流畅。

## 7. 一句话总览

**画布中心 + 冷性中性深色 + shadcn 打底,但画布层(Flow)和环绕工具层(shadcn)共享同一套 design token,绝不两张皮;MVP 只做"拓扑画布 + 右侧属性栏 + AI 提议面板"三块,留六区扩展位;视觉精修和品牌一律推迟到骨架成形之后。**
